# Session — Field-Ops P2.3 write routes (job/task/time/equipment CRUD), 6 slices landed

The field-ops WRITE layer, built on the P2.2 read tabs. Six contained, sequentially-landed slices
turn the read-only Field-Ops dashboards into a system of record: log time, create/close jobs,
add/advance tasks, change equipment readiness, move equipment, log maintenance, and manage the
fleet roster — all capability-gated, send-free, and audit-trailed. CC-authored end-to-end (the
write surface is uniformly security-sensitive — every slice surfaced a real security finding).

## Decisions made during session

- **CC-built all 6 slices, not Le Mans.** The design marked 3/5/6 Le-Mans-eligible, but adjusted to
  CC after the evidence: the security reviewer found a REAL finding in nearly every slice
  (submitted_as normalization #312, unbounded new_client fields #313, missing body-guard #314). The
  "security-critical → CC" principle the design flagged for some slices applies to the whole write
  surface. CC-building small slices off two landed reference impls was faster + surer than
  Le-Mans-plus-heavy-cleanup.
- **The canonical write pattern** (gate → body-guard → referential guards → mutation + audit_log in
  ONE `DB.batch`) was established in Slice 1 (the migration-named reference) + the `audit.ts`
  extraction, then reduced/copied across slices. Integrity-bar tables (time_entries/equipment_logs)
  append-only + amends_uuid; plain tables (jobs/task_assignments/equipment) UPDATE in place; both
  always batch the audit row (conditional `WHERE changes()=1` for guarded no-ops).
- **`submitted_as` is normalized + existence/disabled-checked** (mirroring `/api/submit`) on every
  attributed write — a phantom/unnormalized attribution would corrupt the integrity bar.
- **cap.tasks.own scope = BROAD** (operator-confirmed): any field PM advances any task on the board;
  the 0013 "own assigned" label is imprecise (tidy later, not a code change).
- **Deferred the inspection quick-log** (design's Slice 5): there is no equipment-pre-inspection
  forms catalog to validate `form_code` against — needs the operator to define the forms + codes first.
- **§50 D1-as-writer doctrine** surfaced before Slice 1 landed: P2.3 makes D1 an authoritative writer
  for payroll-grade data without per-entry human approval (send-free, audit-trailed). Built under the
  operator's "proceed" go-ahead; formal Op Stds §50 bump (v18→v19) remains the standing ceremony item.
