# Session — P2.5 Slice 6: portal-owned canonical job number (PR #389)

A pre-cutover revision of P2.5's identity model, prompted by the operator catching a real
two-number mismatch while reviewing the cutover checklist. One PR, four-part verified. Both
adversarial reviewers caught real defects mocks could not — a runbook activation-order BLOCK and a
Worker fail-closed-symptom inaccuracy — both fixed and regression-tested before merge.

## The problem (operator-caught, verified against live HEAD + live Smartsheet)

The shipped P2.5 model gave each job **two identifiers**: the office employee TYPED a `job_id`
(validated; a `JOB-####` shape was REJECTED as `reserved_job_id`), while Smartsheet's AUTO_NUMBER
SEPARATELY assigned the canonical `JOB-####` that the writer omitted-on-create and READ BACK via
`get_row`. A 3-agent investigation workflow + direct Smartsheet column reads confirmed the live
state: safety `ITS_Active_Jobs."Job ID"` is AUTO_NUMBER (`prefix JOB-`, at `JOB-000016`, **1 live
row**); the progress sheet's `Job ID` is TEXT, mirrored from safety; nothing downstream parses the
number or assumes contiguity (week-sheet/Box naming use Project Name; recipient resolution +
intake use `get_job(key)` string-match). So the portal identified a job by its typed key while
every report/sheet/Box folder identified it by `JOB-####` — and the portal couldn't even know the
canonical id until a mirror round-trip. Operator decision (AskUserQuestion): **system-assigned
sequential number, continue the sequence, build now (pre-cutover, daemon OFF, 1 row).**

## Adversarial review — real defects caught + fixed

This is an identity-critical, cross-layer change (a concurrency-sensitive allocator + identity
semantics across D1, both sheets, the writer, and the daemon), so it was CC-led, not delegated, with
both reviewers as DoD.

1. **`ops-stds-enforcer` — BLOCK (§43 activation order).** The Slice-6 activation runbook AND the
   mirrored migration-comment listed the "bump `job_counter`" SQL step BEFORE the step that applies
   `0022` (which CREATES the table) — so running the numbered steps as written fails with
   `no such table: job_counter`. This is the exact forensic-#2 stale-deploy class reproduced inside
   the runbook meant to prevent it. **Fix:** reordered both (retype/add-column → apply 0022 →
   confirm/bump seed → redeploy). **+2 §42 WARN:** the writer's Non-clobber Invariants enumeration
   omitted `Job ID` (now written); an integration-test docstring cited a renamed section. Both fixed.

2. **`portal-worker-security-reviewer` — no BLOCK, sharp W11.** The reviewer built a throwaway D1
   probe and verified that the genuine "0022 not applied" case (table absent) makes D1 THROW
   `no such table` → the global `onError` returns a generic `internal_error` 500 — NOT the
   `counter_unavailable` my code-comment + runbook promised (the clean code only fired for the
   narrower row-deleted sub-case). Still fail-closed (no job, no malformed id — verified), but the
   documented symptom was wrong for the real deploy-order scenario a Tier-2 operator would grep for.
   **Fix:** `allocateJobNumber` now catches any D1 error and collapses to `counter_unavailable`,
   making code and docs true; added a `DROP TABLE job_counter` regression test alongside the existing
   `DELETE FROM` (row-missing) one; corrected the runbook symptom wording (response body, not a log
   line). W5 (sequenced allocate→insert; a seed set too low → 409 burst burning numbers) is an
   accepted, documented tradeoff — no code change; the runbook's seed-check step covers it.

The recurring lesson (forensic #9/#14): adversarial review repeatedly finds what mocks structurally
cannot — here an order-of-operations doc defect that would have re-triggered the lockout class, and a
fail-closed path whose observable symptom didn't match its docs.

## Decisions made during session

1. **Portal owns the number; one identity everywhere.** System-assigned sequential `JOB-######`
   (the office employee types only Project Name), written as `job_id` == `canonical_job_id` ==
   the Smartsheet `Job ID` column on both sheets. Chosen over keeping a human-typed key (operator
   wanted a system "number") and over a year-prefixed scheme (would split the one existing
   `JOB-000016` from new ids).
2. **Counter seed = 16** — the live AUTO_NUMBER max, so the first portal number is `JOB-000017`,
   never colliding with `JOB-000001..000016`. `INSERT OR IGNORE` makes a re-apply a no-op.
3. **Allocate AFTER validation, fail-closed on a missing counter.** A 400/422 never burns a number;
   a missing table/row returns `counter_unavailable` (never a malformed id). A burned-number gap on
   a later insert failure is an accepted tradeoff (AUTO_NUMBER gapped too).
4. **§14 preservation over a clean-schema refactor.** `Portal Job Key` (now redundant with `Job ID`)
   and the `canonical_job_id` mirror machinery (now always-set, idempotent) are kept rather than
   ripped out — the daemon's find-or-create + the down-sync fence keep working unchanged; the
   redundancy is recorded in `docs/tech_debt.md` with a revisit trigger.
5. **Build now, pre-cutover.** The daemon is OFF and the sheet has one row — the cheapest possible
   moment; no dual-number model ever ships, no live data to migrate.
