# Session — P2.5 job-tracker → Smartsheet up-sync (PRs #383, #384, #385, #386, #387)

Made the ITS Portal "new job" form the **authoritative job-creation/lifecycle writer**: a
portal-created job now flows UP into BOTH Active-Jobs systems-of-record (safety
`ITS_Active_Jobs` + progress `ITS_Active_Jobs_Progress`), reversing the old Smartsheet→D1-only
direction. This is the Op Stds v19 §50/§51-blessed SoR write-back (job-tracker → Active-Jobs).
Five PRs landed + four-part verified; **the mirror daemon ships `sync_enabled` OFF** (operator
flips at cutover). Built in **maximum-parallelism mode** per the operator's standing directive —
five build/doc/scaffold tasks ran concurrently in isolated worktrees (each its own fresh
`.venv-wt`), each independently re-gated and adversarially reviewed before merge. Identity stays
typed-key-stable (`origin='portal'` forever, a `Portal Job Key` bridge column carries the
downstream join); a version vector makes a dual-sheet partial failure self-healing.

## Adversarial review — real BLOCKs caught + fixed

Every trust-boundary diff went through its specialist reviewer. Each caught a real defect class
mocks structurally cannot:

1. **#383 (Slice 1) — `portal-worker-security-reviewer` caught 2 BLOCKs.**
   - **W5 cross-origin SoR corruption.** `setLifecycle` and the `/contacts` UPDATE were keyed on
     `job_id` ONLY. A `cap.jobtracker.manage` account could therefore edit an `origin='smartsheet'`
     job's SoR fields, bumping its `mirror_version`/`sync_state='pending'` — but the down-sync only
     reconciles `origin='smartsheet'` jobs FROM Smartsheet, so the corruption would persist
     permanently and the daemon would push the operator's own data back over a fabricated edit.
     **Fix:** all three edit UPDATEs (`setLifecycle`, `/contacts`, `/lifecycle`) are now
     `WHERE job_id=?1 AND origin='portal'`-scoped. Regression-tested; the existing `/close` tests
     had to switch from `seedJob` (smartsheet-origin) to portal-created jobs.
   - **W2 null-body 500.** `/lifecycle` read `body.lifecycle` without a plain-object guard → a null
     body 500'd instead of a clean 400. Guarded + tested.

2. **#387 (Slice 5) — `ops-stds-enforcer` caught a §42 BLOCK.** `active_jobs_writer.py`'s
   module docstring lacked the literal `Invariants` and `Consumers` headings (§42 self-documentation
   requires all four). Fixed; the in-code creds pointer to the runbook was also added.

**Self-inflicted CI miss (lesson logged).** Slice 1's first push failed CI on a TS2571
("Object is of type 'unknown'") in a test-only `(await res.json()).error` access. Root cause: I ran
`npm run typecheck` BEFORE writing the test files, so the test-only unknown-cast wasn't in the
typechecked set. Fixed with `((await res.json()) as any).error`. **Lesson: typecheck AFTER writing
the tests, not before.**

## Decisions made during session

1. **Run the slices in maximum parallelism.** Per the operator's "use as many parallel, workflows,
   and agents possible while still adhering to standards," the five tasks (Slices 1/2/3/5 + the
   §6/A8 doc) ran as concurrent agents in **separate pre-created worktrees with their own fresh
   `.venv-wt`** — never `cp -R .venv` (that repoints the live editable finder; the 2026-06-30
   live-corruption incident). Read-only reviewers (`ops-stds-enforcer`,
   `portal-worker-security-reviewer`) are safe to run in parallel; build agents are NOT co-located.
   Every agent output was independently verified (re-ran the gates, read the diffs, ran the full
   suite) — trust-but-verify, not trust.

2. **Identity is typed-key-stable, not an origin flip.** `job_id` stays the D1 PK; `origin='portal'`
   stays `'portal'` FOREVER (the old migration-0017 origin-flip was a bug, removed). The
   `Portal Job Key` bridge column carries the downstream join; `canonical_job_id` is the safety
   sheet's read-back `JOB-####`. This is why the down-sync needs the canonical-aware pre-pass and
   why every edit route is `origin='portal'`-scoped.

3. **Version vector for dual-sheet partial-failure consistency.** A single `mirror_version` bumped
   on every SoR mutation, plus per-sheet `*_mirrored_version` watermarks; the daemon marks each
   sheet mirrored as its own commit point, and `sync_state` only flips to synced when BOTH
   watermarks catch up. A crash after writing safety but before progress self-heals on the next
   cycle — no double-write, no lost update.

4. **The daemon ships OFF.** `field_ops.fieldops_sync.sync_enabled` defaults OFF; the operator flips
   it at cutover after the secret/Keychain token pair is set and `0021` is applied `--remote`. This
   keeps the write-back capability dark until its prerequisites (Slice 4 column, matching tokens,
   §46/SoR shares) are in place.

5. **`_ENROLLMENT_SUFFIXES += _sync.py` reverted-with-note.** Adding `_sync.py` to the
   capability-gating enrollment suffix list cascaded and flagged the pre-existing
   `shared/picklist_sync.py` as unenrolled, breaking the meta-test. Reverted with an in-code note;
   the proper fix (enroll `picklist_sync.py` first, then add the suffix) is a fast-follow.
